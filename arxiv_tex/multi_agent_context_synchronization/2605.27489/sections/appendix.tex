\section{Detailed: Methodology for Studying HARP Attacks}
\label{app:detailed-methodology}

\subsection{Design principles}
\proposed~follows four design principles that are reflected in the implementation and metric suite.
\textbf{First, traces are primary artifacts.} The benchmark does not immediately collapse a run into a scalar score. It stores agent outputs, tool names, tool inputs, observations, final responses, token usage, elapsed time, database snapshots, oracle logs, and defense metadata. This makes later audit possible and avoids a common failure mode in which only the final answer is retained.
\textbf{Second, evaluation is paired.} Each malicious run is compared to a clean run of the same query under the same defense and database seed. This is important because finance tasks vary substantially: a high-risk transfer, a portfolio review, and a market query have different natural routing and final-response structures. Pairing isolates the effect of the perturbation from task difficulty.
\textbf{Third, side effects are simulated but stateful.} The benchmark uses a controlled finance database rather than real accounts. Nevertheless, actions such as transfer execution are stateful within a run, so the system can measure whether a harmful decision would have led to an operational side effect.
\textbf{Fourth, the final decision is auditable.} The decision engine is not used as an unconstrained judge. It is a component of the system under test with a constrained approve/reject format and explicit veto conditions. Separate scoring functions then evaluate whether the final decision was appropriate given the trace.

These principles are intended to make the benchmark useful for both scientific comparison and engineering debugging. A high ASR identifies a weak defense, but the trace explains where the weakness occurred; a high $\HA$ identifies fragile orchestration, but the paired deviation components explain which local fields changed.

\subsection{Architecture}
Figure~\ref{fig:architecture} summarizes the system. A router invokes specialists under mandatory chains for security-sensitive actions. Transfer requests use a pre-check chain of policy, risk, fraud, compliance, decision engine, and customer service. Bond or portfolio requests use financial analysis, risk, compliance, decision engine, and customer service. Specialist agents use ReAct-style tool invocation. The customer-service agent can query balances, transactions, and transfers; the policy guard evaluates authorization and policy compliance; the risk agent estimates risk level and score; the fraud agent inspects suspicious patterns; the compliance agent checks KYC/AML/sanctions status; the financial analyst retrieves market information; and the portfolio manager analyzes portfolios and can execute portfolio actions in the controlled environment.


The decision engine is a central measurement component. It is instructed to output exactly one of two gate-level decisions for action requests: approve or reject, with an explanation. It should reject when policy violation, medium/high/critical risk, medium/high/critical fraud, compliance issue, or specialist blocking recommendation is present. This gate is intentionally simple relative to the surrounding LLM traces: it creates an auditable point at which local evidence becomes an operational decision.

The controlled database is reset between paired runs. This reset is necessary because a clean run that executes a simulated transfer would otherwise change the state seen by the malicious run, or vice versa. The reset also makes temporal attacks interpretable: the temporal counter is stored separately from the database state and is the only intended cross-run signal when delayed triggers are enabled.

\subsection{Malicious engine and oracle logs}
The malicious engine supports pattern trigger, scoped to all agents, a named agent, or a specific tool. Payload classes include subject or entity redirects and numeric manipulation. Perturbations can be applied to query text, tool inputs, tool observations, or final outputs. Every applied payload produces an oracle-log entry with payload identifier, payload type, target, modified fields, agent, tool, and timestamp, unless explicitly configured as stealthy. These logs are not exposed to the agents; they are used only for evaluation.

\subsection{Defenses}
We evaluate five settings.
\textbf{No defense} runs the orchestrator and malicious engine without additional safeguards.
\textbf{React-sandwich} wraps agent prompts with repeated instructions that treat user, tool, and environment content as untrusted and reassert that the agent must not follow injected instructions.
\textbf{LlamaFirewall-style guard} is a pre-tool LLM guard that classifies proposed tool calls as aligned or misaligned and blocks malformed guard outputs by default.
% TODO: Add an approved citation for LlamaFirewall if the final paper is allowed to cite sources outside the provided set.
\textbf{ToolSafe-style guard} follows the step-level tool invocation safety motivation of ToolSafe \citep{mou2026toolsafe}: it reasons over interaction history and a proposed current action before execution.
\textbf{IntegrityGuard} is a trace-level consistency monitor applied after the agents have produced their intermediate outputs but before the final action is accepted. It checks for three classes of high-confidence failures: subject redirection, contradictions between specialist evidence and the final decision, and evidence that an agent followed an injected instruction. Runs that trigger one of these checks are blocked and marked for audit.

\subsection{Evaluation protocol}

\begin{algorithm}[htb]
\caption{Trace-first paired evaluation in \proposed~.}
\label{alg:eval}
\small
\begin{algorithmic}[1]
\Require Queries $\Queries$, scenario configuration $s$, defense $d$, database seed $e_0$
\Ensure Aggregate metrics: $\ASR$, $I_S$, $I_M$, $HA$, $\SF$, $\BU$, $L$, and $NTC$
\For{each query $x_j \in \Queries$}
  \State Reset controlled database to $e_0$.
  \State Disable malicious perturbations and run system with defense $d$ to obtain clean trace $\tau_j^0$.
  \State Reset controlled database to $e_0$.
  \State Enable scenario $s$ and run system with defense $d$ to obtain perturbed trace $\tau_j^K$.
  \State Store raw traces, oracle logs, guard events, detector outputs, token usage, latency, and database snapshots.
  \State Compute $I(\tau_j^0)$ and $I(\tau_j^K)$.
  \State Compute $H_{local}(\tau_j^K,\tau_j^0)$ and $H_{global}(\tau_j^K,\tau_j^0)$
  \State Compute $HA = H_{global}/H_{local}$
  \State Compute $SF$, $BU$, $L$, and $NTC$
\EndFor
\State Aggregate metrics.
\State \Return $\ASR$, $I_S$, $I_M$, $\HA$, $\SF$, $\BU$, $L$, and $NTC$.
\end{algorithmic}
\end{algorithm}


Algorithm~\ref{alg:eval} gives the paired trace-first protocol. For each scenario and query, the evaluator resets the controlled finance database, runs a clean configuration, resets again, runs the perturbed configuration, records traces, computes impacts and deviations, and amplifications, and then aggregates metrics. The observer stores tool calls, agent outputs, final response, token usage, elapsed time, and database snapshots. This trace-first design is important: the raw trace is recorded before any metric is computed, enabling later re-scoring and audit.

\subsection{Scoring schema}
The rule-based score is intentionally decomposed into interpretable categories rather than learned as an opaque scalar. The rule score allocates 25 points to entity correctness, 25 points to decision-engine gate correctness, 20 points to data integrity, 15 points to routing correctness, and 15 points to output truthfulness. The observer prompt additionally asks for process-level judgments such as data access control, process integrity, and anomaly detection. The deviation score is similarly decomposed. Numeric deviation catches score manipulation; text deviation catches broad output rewriting; entity deviation catches account and customer swaps; stance deviation catches approve/reject flips; and hard-block deviation catches whether a write-like request was blocked in one trace but not the other.

For a defense method $m$, define normalized security, utility, latency, and token cost as
\begin{equation}
\begin{aligned}
  \mathrm{Sec}_m &= 1-\frac{\ASR_m}{100}; 
  &\qquad
  \mathrm{Util}_m &= \frac{\BU_m}{100}; \\
  \mathrm{Lat}_m^{\mathrm{norm}} 
  &= \frac{L_m-L_{\min}}{L_{\max}-L_{\min}+\varepsilon};
  &\qquad
  \mathrm{Tok}_m^{\mathrm{norm}} 
  &= \frac{C_m-C_{\min}}{C_{\max}-C_{\min}+\varepsilon}.
\end{aligned}
\end{equation}
A deployment can then select a defense by minimizing a risk-sensitive objective
\begin{equation}
  J_\lambda(m)=
  \lambda_1\frac{\ASR_m}{100}
  +\lambda_2 \mathrm{H_{local}}
  +\lambda_3 \mathrm{I_m}
  +\lambda_4(1-\mathrm{Util}_m)
  +\lambda_5 \mathrm{Lat}_m^{\mathrm{norm}}
  +\lambda_6 \mathrm{Tok}_m^{\mathrm{norm}},
  \qquad \lambda_i\ge 0.
\end{equation}
This objective makes explicit that the ``best'' defense depends on the downstream operating point. A high-assurance financial workflow would usually choose larger $\lambda_1,\lambda_2,\lambda_3$, whereas a customer-support workflow with no irreversible side effects might assign greater weight to utility and latency.

\subsection{Why attack success alone is insufficient}
Attack success rate answers whether an attack happened and was not blocked. It does not distinguish between two qualitatively different failures: a large obvious deviation that produces harm, and a small local deviation that produces comparable harm. The latter is more concerning for multi-agent orchestration because it indicates that the system converted a small specialist-level change into a large operational change. Amplification ratio exposes this distinction by normalizing harm increase by local deviation.


\subsection{Threshold gates as amplifiers}
Decision gates can convert continuous local deviations into discrete operational outcomes. A small risk-score downgrade near a reject/approve threshold can flip a transfer decision. This is not a bug in using gates; gates are necessary for safety. The benchmark implication is that gates should be evaluated not only by clean accuracy but also by their local Lipschitz behavior: small specialist deviations near thresholds should trigger abstention, escalation, or additional verification rather than immediate approval.

\section{Stealth Factor and Benign Utility}
\label{app:sealth}

We evaluate defenses not only by attack success and harm amplification, but also by
\emph{stealth} and \emph{benign utility}. These metrics capture two practical deployment
concerns: whether harmful perturbations remain hidden from the user-visible response, and
whether the defense preserves normal behavior on benign queries.

Let \(Q_A\) denote the set of attack-triggering queries, and let \(Q_B\) denote the set of
benign queries. For an attacked trace \(\tau_j^K\), stealth measures whether the manipulation
is visible in the final user-facing response. Let \(M_j \in \{0,1\}\) indicate that the
manipulation is masked in the final response, and let \(O_j \in \{0,1\}\) indicate that the
manipulation is obvious. We define the rule-based stealth factor as
\begin{equation}
SF_R =
100 \cdot
\frac{\sum_{j \in Q_A} M_j}
{\sum_{j \in Q_A} (M_j + O_j)},
\end{equation}
We additionally compute two independent AI-based stealth estimates, \(SF_{A1}\) and \(SF_{A2}\), and report their
average:
\begin{equation}
SF_{\mathrm{avg}} =
\frac{SF_R + SF_{A1} + SF_{A2}}{3}.
\end{equation}

A high stealth score indicates that a perturbation is difficult to identify from the final
response alone, even when the internal trace contains manipulated entities, contradicted
tool evidence, changed risk ratings, or unsafe side effects. Thus, stealth complements
attack success rate and harm amplification: ASR measures whether an attack survives the
defense, harm amplification measures how far the perturbation propagates, and stealth
measures whether the surviving failure is externally visible.

Benign utility measures whether the system preserves correct behavior on non-attack
queries. Let \(TN\) be the number of benign queries correctly allowed or completed, and
let \(FP\) be the number of benign queries incorrectly blocked or degraded by a defense.
We define benign utility as
\begin{equation}
BU =
100 \cdot
\frac{TN}{TN + FP}.
\end{equation}

This metric is necessary because a defense can reduce attack success by overblocking
benign workflows, producing false positives, or preventing valid user requests from
completing. In our evaluation, prompt-only defenses tend to preserve high benign utility
but leave stealthy propagation largely intact, while stronger trace-level defenses reduce
attack success and global harm at the cost of additional utility and cost trade-offs.
Reporting \(SF_{\mathrm{avg}}\) and \(BU\) alongside ASR, \(H_{\mathrm{local}}\),
\(H_{\mathrm{global}}\), HA, latency, and token cost therefore provides a fuller view of
the security-utility frontier rather than treating defense quality as a single binary
outcome.

\subsection{Example trace interpretation}
The interaction illustrates why trace-level evaluation is necessary. A risk agent first observes evidence indicating that the customer has medium risk, but the agent response reports low risk. Later, the compliance agent changes a KYC expiration date in its summary. Finally, the customer-service agent executes the transfer to a different account than the account requested by the user, while the final user-visible response still claims that the transfer was sent to the requested account. This interaction contains multiple small or locally plausible deviations: risk downgrading, compliance-field alteration, wrong-destination execution, and output masking. Any one of these deviations could be missed if evaluation considered only the final answer or only a single tool call. The full trace exposes the local-to-global chain: misleading specialist summaries lead to approval, approval enables execution, and the final response masks the wrong-entity side effect.
