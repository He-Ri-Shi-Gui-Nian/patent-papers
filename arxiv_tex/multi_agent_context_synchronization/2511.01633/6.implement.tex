



%\textbf{System Architecture.} 
%The architecture of \projectname is designed with modularity in mind to support flexible customization and extension. Specifically, we separate the system into two core components: (1) the LLM-based agent module and (2) the Graph RAG retriever module, each implemented as independent classes. The agent module supports the instantiation of specialized agents, including \textit{classification}, \textit{thought}, and \textit{action}, using distinct prompt templates. These templates are organized in a dedicated \texttt{custom} directory, allowing users to easily modify reasoning logic or adapt the system to new datasets. The execution logic follows the design outlined in Algorithm~\ref{alg:pipeline}, enabling rapid extension through the addition of new agents or customized control flow.

%\textbf{Parallel Batch Processing.} 
%To support concurrent query processing efficiently, we adopt a thread pool in which each incoming query is assigned to and processed by an independent thread. All threads share the same instances of the LLM-based agent and Graph RAG retriever, which facilitate resource reuse and minimize memory overhead. The vLLM backend is responsible for intelligently batching requests, managing scheduling policies, and executing inference tasks across multiple GPU workers. This ensures that the system achieves both scalability and high efficiency, even when handling a large number of simultaneous requests.


%\textbf{Fault Tolerance.} 
%To ensure robustness, we implement two layers of fault-tolerant handling in \projectname. First, for code execution failures, e.g., due to syntax or semantic errors in code generated by the action agent, we capture both the generated code and the associated error message and feed them back into the LLM for self-correction. This iterative feedback loop allows the model to refine its output and recover from execution errors. Second, for request-level failures, we introduce a timeout and recovery mechanism: if a failure occurs before the first iteration completes, the request is restarted from scratch; if the failure occurs mid-execution, the system resumes from the last successful step by reusing cached KV pairs, avoiding redundant recomputation and improving resilience under load.
