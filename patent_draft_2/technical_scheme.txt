================================================================
面向金融合规领域的多层本体知识图谱GraphRAG系统技术方案
================================================================

一、系统概述

本系统是一个面向金融合规领域的GraphRAG（基于知识图谱的检索增强生成）系统，
由四个核心模块组成：图结构、切片、入图、检索。系统以多层本体知识图谱为知识
表示载体，通过双路径召回与多智能体协同探索机制，为用户查询提供可追溯的合规
证据与回答。

应用场景：金融合规审查。用户提出合规问题（如"证券公司违规代客理财违反了
哪些规定？"），系统从知识图谱中检索相关条款证据，生成附带证据链的回答。


================================================================
二、图结构
================================================================

2.1 节点类型

图中共有四类节点，分为"预置节点"和"入图时创建的节点"两类。

预置节点（存在于空图中，无需随文件入图而创建）：

  (1) 地位类本体节点
      表示法律文件的效力层级。共12类：
      宪法、法律、行政法规、地方性法规、部门规章、地方政府规章、
      行政规范性文件、司法解释、行业自律规则、标准文件、公司章程、
      内部制度。

  (2) 要件类本体节点
      表示法律条款涉及的主体类型和合规要素。共11类：
      O1  监管主体：依法承担金融监管、行政许可、检查、调查、处罚、风险处置等职责的国家机关及其派出机构。
      O2  自律主体：依法或者依章程承担自律管理职责的交易所、登记结算机构、行业协会等组织。
      O3  经营主体：金融机构、支付机构、上市公司、基金管理人、证券期货经营机构、保险机构及其他受监管经营者。
      O4  保护主体：客户、投资者、金融消费者、个人信息主体、存款人、投保人、被保险人等受法律保护的主体。
      O5  合规客体：资金、账户、证券、保险、金融产品、个人信息、重要数据、业务资料等被规范的对象。
      O6  合规行为：开户、交易、销售、披露、数据处理、客户尽调、报告、保存、营销、风控等被规范的行为。
      O7  权利利益：知情权、选择权、公平交易权、财产安全、信息权益、隐私利益、获得赔偿和救济的利益。
      O8  法定义务：尽职调查、适当性管理、信息披露、保密、风险控制、资料保存、报告、整改等作为或不作为要求。
      O9  风险事实：洗钱风险、市场操纵、欺诈销售、数据泄露、信用风险、流动性风险、操作风险、系统性风险等事实状态。
      O10 证据程序：合同、记录、报告、账簿、系统日志、监管报送、检查、调查、听证、复议、诉讼、投诉等材料与程序。
      O11 责任救济：行政处罚、监管措施、纪律处分、民事赔偿、刑事追责、行政复议、行政诉讼、投诉处理、国家赔偿等后果机制。

入图时创建的节点（随法律文件入图而新增）：

  (3) 法律性本体节点
      表示一部具体的法律文件，如《证券法》。
      每个法律文件对应一个节点。

  (4) 内容切片节点
      表示法律文件的具体条款内容。
      每个内容切片对应法律文件中的一个或多个条款。
      每个内容切片携带metadata，包括法律文件名、编、章、节、条、款、项、目等
      层级信息（由LLM从JSON路径中提取）。

2.2 边类型

图中共有四类边：

  (1) 地位类本体之间的边
      有向边。边标签为37个谓词之一（如"不得抵触"、"解释"等）。
      表达不同效力层级之间的法律关系。
      例："法律" --不得抵触--> "宪法"
      例："司法解释" --解释--> "法律"
      这些边在空图中预置。

  (1.1) 37个谓词定义

      1.  最高约束：行设定列的最高合法性边界；列必须在行确定的宪制框架内制定、解释和执行。
      2.  不得抵触：行不得同列相抵触；列对行形成底线限制，行若违反列则应被纠正、撤销或排除适用。
      3.  上位约束：行作为上位规范限制列的制定和适用；列不得突破行设定的权限、义务、责任和程序边界。
      4.  细化执行：行把列的原则性或框架性要求具体化；列通过行进入更具体的行政管理或业务执行场景。
      5.  区域细化执行：行在本行政区域内细化执行列；列通过行转化为属地经营、属地监管或属地合规要求。
      6.  专业细化执行：行在专业监管领域细化执行列；列通过行转化为金融业务、机构监管、市场监管等专业规则。
      7.  按权限细化执行：行在制定机关的法定权限内细化执行列；列通过行进入具体监管口径、操作要求或执行安排。
      8.  区域约束：行在本行政区域和本事项权限内约束列；列在相关区域内不得违反行。
      9.  并行协调：行与列不当然互为上下位；行在自身权限内与列并行适用，列与行发生冲突时应依照法定裁决或适用规则处理。
      10. 提供解释对象：行作为列解释或统一适用的对象；列围绕行确定法律理解、裁判尺度或适用规则。
      11. 统一裁判适用：行统一列在审判或检察适用中的理解；列受行影响，在争议解决和责任认定中产生更确定的法律后果。
      12. 审查适用反馈：行通过裁判规则影响列在个案中的审查和适用；列受行影响，可能被参照、限制或排除适用。
      13. 提供监管背景：行向列提供监管实践和监管政策背景；列在解释或裁判时可把行作为理解金融监管语境的材料。
      14. 监管要求传导：行将监管要求传导给列；列受行影响，需要把监管要求转化为自律规则、章程安排或内部制度。
      15. 承接监管要求：行承接列提出的监管要求；列通过行进入行业自律、公司治理或机构内部控制层面。
      16. 承接细化：行承接列的基本规则并作行业化细化；列通过行形成更贴近市场参与者的行为约束。
      17. 规定采用条件：行规定列何时必须采用、何时可以采用或如何采用；列受行影响，获得强制性、推荐性或引用性的合规地位。
      18. 指引采用：行指引列在监管或业务场景中的采用方式；列受行影响，成为合规检查或业务执行的参考尺度。
      19. 技术支撑：行用技术参数、流程标准或业务标准支撑列的执行；列受行影响，获得可操作的技术尺度。
      20. 提供技术认定依据：行向列提供技术事实或专业判断依据；列可借助行判断系统安全、数据处理、信息披露、业务流程等事项。
      21. 合法性限定：行限定列的合法内容和有效边界；列不得通过章程安排排除、减轻或变更行所确定的强制性要求。
      22. 制度内化要求：行要求列把外部规范转化为内部制度；列受行影响，需要形成岗位职责、审批流程、检查机制和问责规则。
      23. 裁判风险反馈：行把争议解决中的责任认定逻辑反馈给列；列受行影响，需要调整自律规则、标准采用、章程安排或内部制度。
      24. 提供行业事实规范：行向列提供行业惯例、自律义务和市场运行规则；列可借助行理解金融市场主体的注意义务和行为边界。
      25. 引用技术标准：行将列纳入行业自律要求；列受行影响，成为会员或从业人员需要遵守的业务技术尺度。
      26. 自律义务传导：行把行业自律义务传导给列；列受行影响，需要把会员义务、从业义务或交易义务写入章程和内部制度。
      27. 技术控制支撑：行支撑列形成技术控制要求；列受行影响，可以把技术标准转化为系统权限、数据治理、模型管理或操作流程。
      28. 承接自治空间：行在列允许的自治空间内配置治理规则；列通过行落实到机构组织结构、权责分配和内部决策程序。
      29. 提供内部事实依据：行向列提供判断内部权利义务和责任边界的依据；列可据此评价机构内部治理行为。
      30. 嵌入自律义务：行把列规定的自律义务嵌入公司治理结构；列通过行影响公司、股东、董监高或管理层的行为边界。
      31. 嵌入技术要求：行把列规定的技术要求嵌入公司治理或业务治理结构；列通过行影响机构层面的技术合规安排。
      32. 治理统摄：行统摄列的制定方向和权限边界；列需要服从行设定的公司治理结构、授权体系和责任分配。
      33. 内化执行：行把列转化为内部岗位、流程、系统、审查和问责机制；列通过行获得机构内部的执行路径。
      34. 内化裁判风险：行把列体现的裁判尺度转化为内部风险提示和控制措施；列通过行影响机构对诉讼、行政处罚和民事责任的预防。
      35. 内化自律义务：行把列转化为会员管理、员工行为、业务审批和违规问责要求；列通过行获得机构内部执行力。
      36. 内化技术控制：行把列转化为系统配置、数据标准、模型控制、信息安全或操作流程；列通过行进入日常业务控制。
      37. 落实治理安排：行落实列设定的治理结构和授权边界；列通过行进入具体部门职责、审批权限和操作规范。

      无向边。边标签为空。
      一个法律性本体节点仅连接一个地位类本体节点。
      例：《证券法》 --- "法律"
      连接哪一个地位类本体，由人工或LLM在入图时给出。

  (3) 法律性本体与内容切片之间的边
      无向边。边标签为版本信息，格式为：
        "现行版本(自某年月日起)"
        "历史版本(自某年月日至某年月日)"
      由人工在入图时给出。
      同一条法律的不同版本，会连接到各自的内容切片节点，通过不同的版本标签区分。
      一条"历史版本"的边标签可以同时表达法律的施行日期和失效日期。

  (4) 内容切片与要件类本体之间的边
      无向边。边标签为要件类本体的具体实例。
      一个内容切片可以连接到多个要件类本体节点（一对多）。
      例：某条款同时涉及"监管主体"（实例："证监会"）和"经营主体"（实例："证券公司"），
      则创建两条边。
      由LLM在入图时确定每个内容切片对应哪些要件类本体并提取相应实例。

2.3 设计原则

  - 除地位类本体之间的谓词边为有向边外，其余边均为无向边。遵循奥卡姆剃刀原理：
    能用无向边表示的关系，不使用有向边。
  - 司法解释与被解释法律之间不创建显式边。司法解释节点在其metadata中记录
    {对应: <被解释的法律文件>}，检索时通过metadata过滤实现反向查询。
    因为司法解释总量有限（几千条），metadata过滤不会造成性能问题。

2.4 三层本体架构

  - 文档层：法律性本体节点 + 内容切片节点，表示法律文件及其条款内容。
  - 地位层：地位类本体节点 + 谓词边，表示法律效力层级关系。
  - 要件层：要件类本体节点 + 实例标签边，表示法律条款涉及的主体类型与实例。


================================================================
三、切片
================================================================

3.1 流程概述

将法律文件从PDF格式转换为可入图的内容切片列表，经历四个阶段：
PDF → Markdown → JSON → 切片JSON → 入图数据。

3.2 阶段一：文档转换

  输入：PDF格式的法律文件。
  工具：Marker（PDF到Markdown的转换工具）。
  流程：
    1. 使用Marker将PDF转换为中间Markdown文件；
    2. 人工校对Markdown文件的格式，确保标题层级、表格、列表等结构正确。
  输出：经人工校对的Markdown文件。

3.3 阶段二：结构化处理

  输入：Markdown文件。
  流程：按照各级标题（如"第一编"、"第一章"、"第一节"、"第一条"）自动化
  转换为JSON结构。
  输出：JSON文件。示例结构：

  {
    "title": "证券法",
    "sections": [
      {
        "heading": "第一编 总则",
        "sections": [
          {
            "heading": "第一章 基本原则",
            "text": "第一条 为了规范证券发行和交易行为，保护投资者的合法权益，
                     维护社会经济秩序和社会公共利益，促进社会主义市场经济的发展，
                     制定本法。",
            "sections": [...]
          }
        ]
      }
    ]
  }

3.4 阶段三：智能切片

  创建切片Agent，配备以下工具：
    1. 切分(字符串, 切分点)：在指定位置切分字符串。
    2. 合并(字符串1, 字符串2)：合并两个字符串。
    3. 相似度(字符串1, 字符串2)：计算两个字符串的语义相似度。
    4. 校验(原始内容, 切片结果)：检查切片结果拼接后是否等于原始内容（完整性校验）。

  切片Agent的workflow：
    1. 遍历JSON中的所有text属性。
    2. 对每个text属性，Agent自行判断其是否为条款形式：
       - 条款形式（如以"第X条"开头的文本）：按条款切分，每个"条"作为一个切片。
         如果一条内包含多款（"第一款"、"第二款"等），整条作为一个切片，不再拆分。
       - 非条款形式（如概述性文字）：按语义切分，确保每个切片表达完整的意思。
    3. 切片结果附加到原JSON节点的同级位置：
       "chunks": [
         {"id": 1, "chunk": "第一条 为了规范..."},
         {"id": 2, "chunk": "第二条 在中华人民共和国境内..."}
       ]
    4. 每次切片完成后，调用校验工具确认切片拼接后等于原始text内容。

  输出：带有chunks属性的JSON文件。

3.5 阶段四：提取入图数据

  使用脚本从切片JSON中自动化提取两部分信息：
    1. Metadata：提取每个切片在JSON结构中的路径信息（如"证券法 > 第一编 > 第一章 >
       第二条"），然后由LLM判断该路径对应的结构化metadata
       （法律文件名、编、章、节、条、款、项、目等）。
    2. 切片文本：提取每个chunk的文本内容。

  输出：待入图的内容切片列表，每个元素包含文本和metadata。


================================================================
四、入图
================================================================

4.1 空图

在没有任何法律文件入图之前，图中已经存在：
  - 所有地位类本体节点（12类）
  - 所有要件类本体节点（11类，O1-O11）
  - 地位类本体之间的有向谓词边（37个谓词）

4.2 入图流程

一个法律文件入图，需要新增两类节点并建立连接。

步骤一：创建法律性本体节点
  为法律文件创建节点，如<证券法>。

步骤二：连接地位类本体
  由人工或LLM指定该法律文件的效力层级。
  创建无向边：<证券法> --- "法律"。
  一个法律性本体只连一个地位类本体。
  在中国大陆现行法律制度下，不存在一个法律文件对应多个地位类本体的情况。

步骤三：创建内容切片节点并连接法律性本体
  对步骤四（切片阶段）输出的每个chunk，创建一个内容切片节点。
  创建无向边：<证券法> --- <第1条>，边标签为版本信息（人工指定）。
  版本信息格式："现行版本(自2020年3月1日起)" 或 "历史版本(自2005年10月27日至2020年2月29日)"。

步骤四：LLM提取metadata
  将切片JSON的路径信息发送给LLM，由LLM判断每个切片的结构化metadata。

步骤五：连接要件类本体
  由LLM确定每个内容切片对应哪些要件类本体，并提取相应实例。
  创建无向边：<第2条> --- "经营主体"，边标签为"证券公司"。
  一个内容切片可以连接多个要件类本体，每个连接创建一条独立的边。

步骤六：写入metadata
  将metadata写入内容切片节点的属性中，包括：法律文件名、编、章、节、条、款、项、目。
  对于司法解释类文件，在metadata中额外写入{对应: <被解释的法律文件>}。


================================================================
五、检索
================================================================

5.1 总体流程

用户查询q → 实体提取 → 双路径召回 → 多智能体探索 → 会议主持过滤 →
审计兜底 → 证据输出 → LLM生成回答。

5.2 实体提取

  LLM从用户查询q中提取：
    - 涉及的法律文件（可为多个）
    - 要件类本体及其对应实例（可为多个）
  例：查询"证券公司违规代客理财违反了哪些规定？"
    提取结果可能为：
      法律文件：无（用户未指定具体法律）
      要件本体-实例对：(经营主体, 证券公司)

5.3 双路径召回

  路径一：结构化召回
    1. 候选节点生成：
       第一部分：LLM提取的法律文件节点所连接的所有内容切片。
       第二部分：LLM提取的要件类本体节点，从边中筛选出匹配对应实例的边，
                沿这些边反向找到连接的内容切片节点。
    2. 在候选节点集合中，使用稠密检索（dense retrieval）与稀疏检索（如BM25）
       相结合的混合检索方式，返回top-k个内容切片。

  路径二：语义召回
    1. LLM对用户查询q进行多版本改写，得到(q_1, q_2, ..., q_m)。
    2. (q, q_1, ..., q_m)共(m+1)个查询，逐个在所有内容切片节点中做稠密检索，
       得到(m+1)个top-k。
    3. 对每个top-k分别使用Cross Encoder进行重排序。
    4. 使用RRF（Reciprocal Rank Fusion）算法融合(m+1)个重排序后的结果，
       得到最终的top-k。

  合并：
    路径一的k个节点与路径二的k个节点合并去重，得到至多2k个节点，记为n个节点。

5.4 多智能体探索

  创建n个检索Agent，每个Agent对应一个出发节点（即召回的内容切片）。
  所有Agent共享以下工具集：
    Tool 0：查看节点的metadata。
    Tool 1：查看邻节点与对应的边标签。
    Tool 2：记录路径。
    Tool 3：在所有节点中做稠密或稀疏搜索。
    Tool 4：中止探索。

  每个Agent的执行流程：

  第一阶段：时间适用性检查
    Agent判断出发节点是否在时间上适用于用户查询。
    依据：出发节点与法律性本体之间边的版本标签（如"现行版本(自2020年3月1日起)"
    或"历史版本(自2005年至2020年)"）。
    如果不适用（如用户问的是2018年的情况，但出发节点属于2020年修订版）：
      Agent寻找同一法律文件的时间适用版本，从该版本所连接的内容切片中找到与
      出发节点对应的内容切片，重设出发节点。
    如果适用：直接进入下一阶段。

  第二阶段：上下文补全计划
    Agent被赋予"补全出发节点上下文"的任务。
    Agent自行判断出发节点的内容切片缺少哪些上下文（如引用了未包含的条款、
    涉及未列出的主体、缺少处罚条款等）。
    Agent输出一个上下文补全计划，明确需要补全哪些上下文。

  第三阶段：Tool Call执行
    Agent根据计划，调用工具收集上下文信息。
    每次调用后，Agent更新已收集的上下文和探索路径。

  终止条件（满足任一即终止）：
    - Agent判断上下文已补全：输出(出发节点, 上下文补全计划, 探索路径, 推理结果)。
    - 达到最大Tool Call次数：输出(出发节点, 上下文补全计划, 探索路径, 已补全的上下文)。

5.5 会议主持

  在所有检索Agent都终止探索之后，创建会议主持Agent。

  会议主持Agent的职责：
    1. 听取每个检索Agent的输出。
    2. 判断每个出发节点在法律范围意义上是否适用于用户查询q。
       注意：这是法律范围适用性判断（如"这条法律是否与证券公司代客理财有关"），
       与前面Agent自行完成的时间适用性检查是两个独立判断。
       范围适用性的判断标准由会议主持Agent自行决定。
    3. 筛掉法律范围不适用的检索Agent。
    4. 对剩余适用的检索Agent中，尚未完成上下文补全计划的，重置Tool Call次数，
       让它们再次进行探索。

5.6 审计

  如果再次探索后，仍有检索Agent触及最大Tool Call次数且未完成上下文补全计划：
    对应每个这样的Agent，创建一个审计Agent。

  审计Agent的职责：
    1. 判断对应的检索Agent是否在"兜圈子"（判断标准由审计Agent自由决定，
       如重复访问相同节点、工具调用无新信息等）。
    2. 兜圈子的：终止探索，输出(出发节点, 上下文补全计划, 探索路径, 已补全的上下文)。
    3. 仍有探索价值的：再次重置Tool Call次数，审计Agent配合各自的检索Agent继续迭代。
    4. 重复上述过程，直到不存在仍需继续探索的检索Agent为止。

5.7 证据输出

  所有Agent终止后，最终得到x个结构化证据：
    (出发节点, 上下文补全计划, 探索路径, 计划完成情况)

  其中 x ≤ n（因为会议主持阶段筛掉了法律范围不适用的Agent所对应的检索Agent）。

5.8 回答生成

  将x个结构化证据作为上下文，喂给LLM。
  LLM基于这些证据生成最终回答。
  每条证据都有明确的来源节点和探索路径，回答具备可追溯性。


================================================================
六、系统特征总结
================================================================

6.1 知识表示方面

  - 三层本体架构（文档层+地位层+要件层）统一建模法律文件的层级结构、
    效力关系和主体类型。
  - 版本标签精确表达法律的时间适用性，支持"施行日期"和"失效日期"的表达。
  - 要件本体的实例标签表达条款涉及的具体主体。
  - 地位类本体之间的谓词边表达法律效力层级关系。

6.2 检索方面

  - 双路径召回：结构化路径利用图结构做精确召回，语义路径通过多版本改写+
    Cross Encoder+RRF做宽泛召回，两者互补。
  - 多智能体并行探索：每个候选节点独立分配Agent，并行执行探索。
  - 时间适用性自动检查：Agent在探索前自动判断版本时效。
  - 上下文补全机制：Agent自主判断缺失上下文并制定补全计划。
  - 分层质量控制：会议主持Agent做粗筛（范围适用性），审计Agent做细筛
    （兜圈子检测），两层质控确保探索效率与质量。

6.3 可解释性方面

  - 每条证据附带完整的探索路径和上下文补全情况。
  - 回答可追溯到具体的法律条款及其关联条款。
  - 满足金融合规审计的可追溯性要求。
