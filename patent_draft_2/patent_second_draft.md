# 发明专利申请书（二稿）

## 一、发明名称

**一种面向金融合规领域的多层本体知识图谱构建与智能检索方法**

## 二、技术领域

本发明涉及人工智能与知识管理技术领域，具体涉及一种面向金融合规领域的多层本体知识图谱构建方法与基于该图谱的智能检索方法，属于知识图谱、检索增强生成（RAG）与多智能体协同技术的交叉领域。

## 三、背景技术

### 3.1 金融合规领域文档检索的技术需求

金融合规工作要求从业人员能够快速、准确地检索和理解大量法律法规、监管规定、行业标准等规范性文件。这些文件具有以下特征：

1. **层级复杂**：中国法律体系包含宪法、法律、行政法规、部门规章、地方性法规、司法解释等多个层级，不同层级之间存在"不得抵触"等效力关系；
2. **版本众多**：同一法律文件可能经历多次修订，存在现行版本与历史版本的区分，不同版本的适用时间范围不同；
3. **主体多样**：法律条款涉及监管主体（如证监会）、自律主体（如证券业协会）、经营主体（如证券公司）等多种类型；
4. **结构规范**：法律文件采用"编-章-节-条-款-项-目"的标准化层级结构；
5. **关联紧密**：法律条文之间存在引用、解释、补充等复杂关系。

### 3.2 现有技术方案及其局限性

通过对近年来相关技术文献的检索与分析，现有金融合规文档检索方案主要包括以下几类：

#### 3.2.1 基于关键词的传统检索

**技术方案**：采用BM25、TF-IDF等传统信息检索方法，通过关键词匹配实现文档检索。

**局限性**：
- 无法理解语义相似性，对同义词、近义表达识别能力差；
- 无法利用法律文件之间的结构化关系（如层级关系、版本关系）；
- 检索结果缺乏可解释性，用户难以理解为何返回特定结果；
- 无法处理复杂的多跳推理查询（如"某行为的监管主体是谁"需要跨越多个条款）。

#### 3.2.2 基于大语言模型的问答系统

**技术方案**：直接使用GPT、Claude等大语言模型回答合规问题，或将法律文件作为上下文输入LLM。

**局限性**：
- **幻觉问题**：LLM可能生成看似合理但实际错误的回答，在合规领域风险极高；
- **上下文窗口限制**：法律文件通常较长，无法完整放入LLM的上下文窗口；
- **知识过时**：LLM的训练数据存在截止日期，无法保证回答基于最新修订的法规；
- **缺乏可追溯性**：无法提供回答所依据的具体条款，难以进行合规审计；
- **版本混乱**：无法区分现行有效版本与历史版本，可能导致适用错误。

#### 3.2.3 检索增强生成（RAG）系统

**技术方案**：将法律文件切分为多个文档片段，通过向量检索召回相关片段，再交由LLM生成回答。

**局限性**：
- **切片粒度问题**：简单的固定长度切片或按段落切片会破坏法律条款的完整性（如将一条法律分成多个片段）；
- **结构信息丢失**：切片后的文档片段失去了与法律层级结构的关联（如某条属于哪一章、哪一编）；
- **版本管理缺失**：无法区分同一法律的不同版本，可能导致召回的内容来自已失效的旧版本；
- **层级关系缺失**：无法表达法律之间的效力层级关系（如行政法规不得抵触法律）；
- **主体识别困难**：难以识别法律条款中涉及的各类主体（监管机构、自律组织、经营机构）；
- **多版本改写不足**：单一查询的检索方式难以覆盖用户意图的多种表达方式。

#### 3.2.4 基于知识图谱的检索系统

**技术方案**：构建法律知识图谱，利用图结构表达法律实体及其关系，支持结构化检索与推理。

**现有方案的局限性**：
- **本体设计不完整**：现有法律知识图谱通常只关注法律文件本身，缺乏对法律效力层级、适用主体等多维本体的统一建模；
- **版本管理简单**：多数方案仅记录法律的发布日期，缺乏对"现行有效时间段"的精确表达；
- **图结构扁平化**：将法律文件、条款、主体等不同层次的信息放在同一层面，缺乏清晰的本体层级划分；
- **检索策略单一**：通常采用基于图遍历的检索，缺乏多路径召回与智能体协同探索机制；
- **时间适用性检查缺失**：检索时未考虑法律版本的时间适用性，可能返回已失效的条款；
- **质量控制不足**：缺乏对检索过程的审计机制，无法识别检索过程中的无效探索。

#### 3.2.5 多智能体协同系统

**技术方案**：使用多个智能体协作完成复杂任务，如一个智能体负责检索、一个负责推理、一个负责验证。

**现有方案的局限性**：
- **角色分工粗放**：通常仅区分检索、推理、验证等粗粒度角色，缺乏细粒度的质量控制机制；
- **缺乏审计机制**：智能体执行过程中的错误难以被发现和纠正；
- **探索效率低**：多个智能体并行探索时，缺乏协调机制，容易出现重复探索或无效探索；
- **上下文补全不完整**：智能体在检索过程中缺乏主动判断"上下文是否完整"的能力。

### 3.3 技术问题的总结

综合上述分析，现有金融合规文档检索方案存在以下核心技术问题：

1. **知识表示问题**：缺乏对法律文件多层本体结构（文件本体、地位本体、要件本体）的统一建模方法；
2. **版本管理问题**：缺乏对法律文件版本的时间适用性的精确表达与检索时的自动检查机制；
3. **检索策略问题**：缺乏结合结构化召回与语义召回的双路径检索方法；
4. **探索效率问题**：缺乏多智能体协同探索与质量审计机制，导致检索效率低、质量不可控；
5. **可解释性问题**：检索结果缺乏可追溯的证据链，无法满足合规审计要求。

## 四、发明内容

### 4.1 发明目的

本发明的目的在于提供一种面向金融合规领域的多层本体知识图谱构建与智能检索方法，解决现有技术中知识表示不完整、版本管理缺失、检索策略单一、探索效率低下、可解释性不足等技术问题。

### 4.2 技术方案

为实现上述目的，本发明采用如下技术方案：

#### 4.2.1 多层本体知识图谱结构

本发明构建的知识图谱包含四类节点和四类边，形成三层本体架构。

**节点类型**：

1. **地位类本体节点**（预置）：表示法律文件的效力层级，包括宪法、法律、行政法规、地方性法规、部门规章、地方政府规章、行政规范性文件、司法解释、行业自律规则、标准文件、公司章程、内部制度共12类；

2. **要件类本体节点**（预置）：表示法律条款涉及的主体类型，包括但不限于监管主体、自律主体、经营主体等；

3. **法律性本体节点**（入图时创建）：表示具体的法律文件，如《证券法》、《公司法》等，每个法律文件对应一个节点；

4. **内容切片节点**（入图时创建）：表示法律文件的具体条款内容，每个内容切片对应法律文件中的一个或多个条款。

**边类型**：

1. **地位类本体之间的边**：有向边，边标签为37个谓词之一（如"不得抵触"、"解释"等），表达不同效力层级之间的法律关系；

2. **法律性本体与地位类本体之间的边**：无向边，边标签为空，表示法律文件的效力层级归属，每个法律性本体节点仅连接一个地位类本体节点；

3. **法律性本体与内容切片之间的边**：无向边，边标签为版本信息，格式为"现行版本(自某年月日起)"或"历史版本(自某年月日至某年月日)"，表示内容切片所属的法律版本及其适用时间范围；

4. **内容切片与要件类本体之间的边**：无向边，边标签为要件类本体的具体实例（如"证监会"、"证券公司"），表示该条款涉及的主体类型及其具体实例，一个内容切片可连接多个要件类本体节点。

**三层本体架构**：

- **文档层**：法律性本体节点与内容切片节点，表示法律文件及其内容；
- **地位层**：地位类本体节点及其之间的谓词边，表示法律效力层级关系；
- **要件层**：要件类本体节点及其与内容切片的连接，表示法律条款涉及的主体类型。

#### 4.2.2 法律文件切片与入图方法

本发明采用如下方法将法律文件转换为知识图谱中的节点和边：

**步骤1：文档转换**

将PDF格式的法律文件转换为Markdown格式：
- 使用Marker工具进行PDF到Markdown的初步转换；
- 人工校对转换后的Markdown文件，确保格式正确、内容完整。

**步骤2：结构化处理**

将Markdown文件按照标题层级（如"第一编"、"第一章"、"第一条"）转换为JSON结构：
```json
{
  "title": "证券法",
  "sections": [
    {
      "heading": "第一章 总则",
      "text": "第一条 为了规范...",
      "sections": [
        {
          "heading": "第一节 基本原则",
          "text": "第二条 在中华人民共和国境内..."
        }
      ]
    }
  ]
}
```

**步骤3：智能切片**

创建切片智能体（Chunk Agent），配备以下工具：
- **切分工具**：`split(text, split_points)`，在指定位置切分字符串；
- **合并工具**：`merge(text1, text2)`，合并两个字符串；
- **相似度工具**：`similarity(text1, text2)`，计算两个字符串的语义相似度；
- **校验工具**：`verify(original, chunks)`，检查切片结果拼接后是否等于原始内容。

切片智能体的工作流程：
1. 遍历JSON结构中的所有`text`属性；
2. 对每个`text`，判断其是否为条款形式：
   - 若为条款形式（如以"第X条"开头），按条款切分，每个"条"作为一个切片；
   - 若为非条款形式（如概述性文字），按语义切分，确保每个切片表达完整的意思；
3. 将切片结果附加到原JSON节点：
```json
{
  "heading": "第一章 总则",
  "text": "第一条 为了规范...",
  "chunks": [
    {"id": 1, "chunk": "第一条 为了规范证券发行和交易行为..."},
    {"id": 2, "chunk": "第二条 在中华人民共和国境内..."}
  ]
}
```
4. 使用校验工具确保所有切片拼接后等于原始`text`内容。

**步骤4：Metadata提取**

使用脚本提取每个切片在JSON结构中的路径信息（如"证券法 > 第一编 总则 > 第一章 基本原则 > 第二条"），然后由LLM判断该路径对应的metadata（如法律文件名、编、章、条等）。

**步骤5：建立图谱连接**

对于每个法律文件的入图：
1. 创建法律性本体节点；
2. 由人工或LLM指定该法律性本体节点应连接的地位类本体节点（如《证券法》连接"法律"）；
3. 由人工指定该法律文件的版本信息（如"现行版本(自2020年3月1日起)"）；
4. 为每个内容切片创建节点，并建立与法律性本体节点的连接（带版本标签的边）；
5. 由LLM识别每个内容切片涉及的主体类型，并提取具体实例；
6. 建立内容切片节点与要件类本体节点的连接（带实例标签的边）。

#### 4.2.3 双路径智能检索方法

当用户提交查询时，系统采用双路径召回策略：

**路径1：结构化召回**

1. **实体提取**：LLM从用户查询中提取涉及的法律文件名（可为多个）和要件本体-实例对（可为多个）；
2. **候选节点生成**：
   - 第一部分：法律文件节点连接的所有内容切片；
   - 第二部分：要件类本体节点通过边标签匹配实例后连接的内容切片；
3. **混合检索**：在候选节点集合中，使用稠密检索（dense retrieval）与稀疏检索（sparse retrieval，如BM25）相结合的方式，返回top-k个相关内容切片。

**路径2：语义召回**

1. **多版本改写**：LLM对用户查询进行多版本改写，生成(m+1)个查询变体（包括原始查询）；
2. **逐个检索**：对每个查询变体，在所有内容切片中进行稠密检索，得到(m+1)个top-k结果；
3. **重排序**：对每个top-k结果分别使用Cross Encoder进行重排序；
4. **融合**：使用Reciprocal Rank Fusion (RRF)算法融合(m+1)个重排序后的结果，得到最终的top-k。

**合并去重**

将路径1和路径2的top-k结果合并，去重后得到n个候选节点（n ≤ 2k）。

#### 4.2.4 多智能体协同探索机制

对每个候选节点，创建一个检索智能体（Retrieval Agent），配备以下工具：
- **查看metadata工具**：`get_metadata(node)`，获取节点的元数据信息；
- **查看邻节点工具**：`get_neighbors(node)`，获取节点的邻节点及对应的边标签；
- **记录路径工具**：`record_path(node)`，记录当前探索路径；
- **全局搜索工具**：`search(query)`，在所有节点中进行稠密或稀疏搜索；
- **中止探索工具**：`stop()`，终止当前智能体的探索。

**智能体探索流程**：

1. **时间适用性检查**：
   - 智能体检查出发节点所属法律文件的版本信息（边标签）；
   - 若该版本不适用于用户查询涉及的时间点，智能体寻找同一法律文件的时间适用版本，并将出发节点重置为该版本对应的内容切片。

2. **上下文补全计划**：
   - 智能体判断出发节点的内容是否完整（如是否引用了未包含的条款、是否涉及未列出的主体等）；
   - 智能体制定上下文补全计划，明确需要补充哪些上下文信息。

3. **工具调用执行**：
   - 智能体根据计划，调用工具收集上下文信息；
   - 每次工具调用后，智能体更新已收集的上下文和探索路径。

4. **终止条件**：
   - 若智能体判断上下文已补全，终止探索，输出：出发节点、上下文补全计划、探索路径、计划完成情况；
   - 若达到最大工具调用次数，终止探索，输出已收集的信息。

#### 4.2.5 会议主持与质量审计机制

**会议主持智能体（Meeting Agent）**：
1. 听取所有检索智能体的输出；
2. 判断每个出发节点是否在**法律范围意义上**适用于用户查询（与时间适用性检查不同，这是判断法律领域是否相关）；
3. 过滤掉范围不适用的智能体；
4. 对剩余的智能体，重置工具调用次数，允许继续探索。

**审计智能体（Audit Agent）**：
1. 对于再次达到最大工具调用次数且仍未完成上下文补全的智能体，创建对应的审计智能体；
2. 审计智能体判断检索智能体是否在"兜圈子"（如重复访问相同节点、工具调用无新信息等）；
3. 对于兜圈子的智能体，终止其探索；
4. 对于仍有探索价值的智能体，配合其继续探索，重置工具调用次数；
5. 重复上述过程，直到所有智能体终止探索。

#### 4.2.6 证据生成与回答生成

**证据生成**：
- 最终得到x个结构化证据（x ≤ n，因为筛掉了范围不适用的智能体）；
- 每个证据包含：出发节点、上下文补全计划、探索路径、计划完成情况。

**回答生成**：
- LLM基于x个结构化证据生成最终回答；
- 回答中引用具体证据，提供可追溯的合规建议。

### 4.3 有益效果

本发明的有益效果包括：

1. **知识表示完整性**：通过三层本体架构（文档层、地位层、要件层），全面表达法律文件的层级结构、效力关系和主体类型；

2. **版本管理精确性**：通过边标签精确表达法律版本的时间适用性，检索时自动检查版本适用性，避免适用已失效的法规；

3. **检索策略多样性**：双路径召回结合了结构化检索的精确性与语义检索的灵活性，提高召回率和准确率；

4. **探索效率高**：多智能体协同探索机制支持并行处理，会议主持与审计机制确保探索质量，避免无效探索；

5. **可解释性强**：每个回答都附带完整的证据链（出发节点、探索路径、上下文补全情况），满足合规审计要求；

6. **可扩展性好**：本体结构清晰，新增法律文件时只需按流程入图，不影响现有结构。

## 五、具体实施方式

### 5.1 系统整体架构

本发明的系统整体架构参见附图1（system_architecture.graphml），包括以下模块：

1. **文档处理模块**：负责将PDF法律文件转换为结构化知识图谱节点；
2. **知识图谱模块**：存储和管理多层本体知识图谱；
3. **检索模块**：实现双路径召回与多智能体协同探索；
4. **生成模块**：基于检索证据生成最终回答。

### 5.2 知识图谱结构

知识图谱的详细结构参见附图2（knowledge_graph_structure.graphml），展示了：

### 5.4 金融合规应用场景示例

**场景**：证券公司合规人员咨询"证券公司违规代客理财违反了哪些规定？"

**系统处理流程**：

1. **实体提取**：LLM提取出法律文件（空，因为用户未指定具体法律）和要件本体-实例对（经营主体-证券公司）；

2. **双路径召回**：
   - 路径1：通过"经营主体-证券公司"找到所有涉及证券公司的内容切片；
   - 路径2：对查询进行多版本改写（如"证券公司代客理财违规"、"证券经纪业务禁止行为"等），在所有内容切片中检索；

3. **智能体探索**：
   - 每个召回的内容切片分配一个智能体；
   - 智能体检查时间适用性（如《证券法》2020年修订版是否现行有效）；
   - 智能体判断上下文是否完整（如是否包含处罚条款、是否涉及监管主体）；
   - 智能体通过工具调用补充上下文（如沿边找到《证券法》第140条关于禁止代客理财的规定）；

4. **会议主持**：
   - 判断哪些法律文件与"证券公司代客理财"主题相关；
   - 过滤掉不相关的智能体（如涉及基金管理公司的条款）；

5. **证据生成**：
   - 输出多个结构化证据，如：
     - 出发节点：《证券法》第140条
     - 上下文补全计划：补充处罚条款、监管主体
     - 探索路径：第140条 → 第205条（处罚）→ 证监会（监管主体）
     - 完成情况：已完成

6. **回答生成**：
   - LLM基于证据生成回答："根据《证券法》第140条，证券公司不得接受客户的全权委托而决定证券买卖...违反该规定的，根据第205条，由证监会责令改正，没收违法所得，并处以违法所得一倍以上五倍以下的罚款..."

## 六、权利要求书

1. 一种面向金融合规领域的多层本体知识图谱构建方法，其特征在于，所述知识图谱包含以下节点类型：
   - 地位类本体节点，表示法律文件的效力层级；
   - 要件类本体节点，表示法律条款涉及的主体类型；
   - 法律性本体节点，表示具体的法律文件；
   - 内容切片节点，表示法律文件的具体条款内容；
   
   所述知识图谱包含以下边类型：
   - 地位类本体节点之间的有向边，边标签为谓词，表达不同效力层级之间的法律关系；
   - 法律性本体节点与地位类本体节点之间的无向边，表示法律文件的效力层级归属；
   - 法律性本体节点与内容切片节点之间的无向边，边标签为版本信息，表示内容切片所属的法律版本及其适用时间范围；
   - 内容切片节点与要件类本体节点之间的无向边，边标签为要件类本体的具体实例，表示该条款涉及的主体类型及其具体实例。

3. 根据权利要求1所述的方法，其特征在于，所述版本信息的格式为"现行版本(自某年月日起)"或"历史版本(自某年月日至某年月日)"。

4. 根据权利要求1所述的方法，其特征在于，所述要件类本体节点包括但不限于：监管主体、自律主体、经营主体。

5. 一种面向金融合规领域的智能检索方法，基于权利要求1-4任一项所述的多层本体知识图谱，其特征在于，包括以下步骤：
   
   步骤S1：接收用户查询，由大语言模型从查询中提取涉及的法律文件名和要件本体-实例对；
   
   步骤S2：采用双路径召回策略生成候选节点集合：
   - 路径1（结构化召回）：获取法律文件节点连接的内容切片和要件类本体节点通过边标签匹配实例后连接的内容切片，使用稠密检索与稀疏检索相结合的方式返回top-k结果；
   - 路径2（语义召回）：对用户查询进行多版本改写，对每个查询变体在所有内容切片中进行稠密检索，使用交叉编码器重排序后通过倒数秩融合算法合并，得到top-k结果；
   - 合并去重路径1和路径2的结果，得到n个候选节点；
   
   步骤S3：为每个候选节点创建检索智能体，每个智能体配备查看元数据、查看邻节点、记录路径、全局搜索、中止探索等工具，执行以下探索流程：
   - 检查出发节点的时间适用性，若不适用则重置出发节点；
   - 制定上下文补全计划；
   - 调用工具执行探索，收集上下文信息；
   - 满足终止条件时输出结构化证据；
   
   步骤S4：会议主持智能体判断出发节点的法律范围适用性，过滤不适用的智能体，重置剩余智能体的工具调用次数；
   
   步骤S5：审计智能体判断检索智能体是否在兜圈子，终止无效探索或配合继续探索；
   
   步骤S6：基于所有智能体输出的结构化证据，由大语言模型生成最终回答。

6. 根据权利要求5所述的方法，其特征在于，所述结构化证据包括：出发节点、上下文补全计划、探索路径、计划完成情况。

8. 根据权利要求5所述的方法，其特征在于，所述法律范围适用性判断是指：判断出发节点所涉及的法律领域是否与用户查询的主题相关。

10. 一种面向金融合规领域的法律文件切片方法，其特征在于，包括以下步骤：

## 八、摘要

本发明公开了一种面向金融合规领域的多层本体知识图谱构建与智能检索方法。该方法构建了包含地位类本体、要件类本体、法律性本体和内容切片四类节点的知识图谱，通过版本标签和实例标签精确表达法律文件的层级关系、时间适用性和主体类型。在检索阶段，采用双路径召回策略结合结构化检索与语义检索，并通过多智能体协同探索机制实现上下文补全，会议主持与审计机制确保检索质量。最终生成的回答附带完整的证据链，满足合规审计要求。本发明解决了现有金融合规文档检索方案中知识表示不完整、版本管理缺失、检索策略单一、可解释性不足等问题。
