一、Task 1：LLM GraphRAG / KG-RAG 方向的主要学术观点

1. GraphRAG 的核心不是“把知识图谱换成向量”，而是把外部知识组织成可遍历、可约束、可解释的结构化检索索引。

这些论文普遍把 GraphRAG 看作 RAG 的结构化扩展：普通 RAG 检索文本块，GraphRAG 检索实体、关系、路径、社区、子图、超边或图摘要。LLM 仍负责理解问题和生成答案，图结构负责把分散证据连接起来。

因此，这个方向与传统 KG embedding 不同。它关注的是：
问题如何映射到图上的检索任务；
图上的哪些节点、边、路径、社区或子图应进入上下文；
这些结构化证据如何帮助 LLM 生成更可靠答案。

2. “GraphRAG 一定优于 RAG”不是共识；更准确的观点是：GraphRAG 只在特定问题类型上有优势。

多篇评测类论文反复强调，GraphRAG 的收益取决于问题复杂度和语料结构。

普通向量 RAG 在简单事实查询、局部文本匹配、答案集中出现在少数文本块中的任务上仍然很强，且成本低。
GraphRAG 的优势主要出现在：
多跳问答；
跨文档证据整合；
全局总结；
领域概念之间存在明确层级、因果、时间、法规或实体关系的场景；
需要解释检索路径或证据来源的高风险应用。

这形成了一个重要学术观点：GraphRAG 不是 RAG 的无条件替代品，而是面向结构性问题的检索增强范式。

3. 该方向的第一个技术焦点是“图怎么建”：从通用 LLM 抽三元组，转向领域约束、任务驱动和可增量维护。

早期 GraphRAG 通常用 LLM 从文本块中抽实体和关系，再做社区发现或图检索。后续论文认为这种建图方式有明显问题：
抽取容易漏掉跨 chunk 的关系；
实体和关系类型不受约束，导致领域语义漂移；
静态图与下游问答目标脱节；
图构建成本高，且更新困难。

对应技术方案包括：
用 ontology / schema 约束抽取，例如 OMD-GraphRAG、法律规范 Graph RAG、DSRAG；
构造领域专用图，例如 UMLS 生物医学图、EHR schema 图、交通安全多模态图、等离子体文献图、客服工单图；
把图分成概念图和实例图，例如 DSRAG；
用 tag hierarchy 代替昂贵社区抽取，例如 TagRAG；
用 GNN 或自监督图破坏发现缺失跨 chunk 关系，例如 Cross-Chunk Graph Augmentation；
用强化学习直接优化“对下游 RAG 有用的图”，例如 AutoGraph-R1；
按查询即时构造证据图，而不是完全依赖预构建静态图，例如 Relink。

核心观点是：GraphRAG 的图不应只追求“看起来像知识图谱”，而应服务于检索和生成任务。

4. 第二个技术焦点是“图怎么组织”：社区、k-core、层级、超图、标签链都在竞争。

Microsoft GraphRAG 一类方法强调社区发现和社区摘要，适合全局 sensemaking 和查询聚焦总结。但 Core-based Hierarchies 指出 Leiden 社区在稀疏图上不稳定，可能产生不可复现的近似最优划分，因此提出 k-core 分解来获得确定性、密度感知的层级。

其他论文提出不同组织方式：
HyperGraphPro 和 Graph-R1 用超图/知识超图表达多元关系；
EHRAG 用语义超边连接原本不共现但语义相关的实体；
TagRAG 用标签链组织领域知识；
法律 Graph RAG 用时间版本、法律行为、条款层级组织图；
SafeDriveRAG 和 DSRAG 用多模态/领域结构组织证据。

学术分歧在于：图结构应更偏通用社区摘要，还是更偏领域本体、任务结构和确定性索引。

5. 第三个技术焦点是“怎么检索”：研究对象已从 top-k 文本块变成路径、子图、证据链和检索轨迹。

该方向大量论文认为，GraphRAG 的真正价值不在“多拿一些文本”，而在“拿到结构上连贯的证据”。

常见检索方案包括：
基于路径的检索：PathISE、StepChain GraphRAG、PROPEX-RAG、HELP；
基于子图的检索：STEM、BubbleRAG、BYOKG-RAG、SubgraphRAG、GraphSearch；
基于传播的检索：SA-RAG 使用 spreading activation；
基于 PPR / random walk / graph traversal 的检索；
基于 GNN 或 Graph Foundation Model 的检索：GFM-RAG、Graph retriever 类方法；
语义检索与图检索混合：LightRAG 风格、Hybrid GraphRAG、ORAN benchmark、ROGRAG；
LLM 代理式图探索：Graph-R1、GraphSearch、PersonalAI 2.0、Graph Counselor。

一个共同观点是：纯图遍历容易受噪声和缺边影响，纯向量检索又容易丢失结构。因此更稳健的方向是“语义相似度 + 图结构约束 + 必要时由 LLM 规划”。

6. 第四个技术焦点是“是否要迭代/代理式检索”：趋势是肯定的，但风险也更大。

许多新论文把 GraphRAG 从一次性检索改造成多轮过程：
先理解问题；
再定位实体或关系；
然后扩展邻居、路径或子图；
再检查证据是否足够；
最后生成答案。

Graph-R1、GraphRAG-R1、GraphSearch、PersonalAI 2.0、Relink、Graph Counselor 等都属于这一趋势。

它们的学术主张是：复杂问题不能靠一次 top-k 检索解决，检索过程本身应当像推理过程一样逐步展开。

但另一组论文提醒：迭代检索可能引入检索漂移、错误路径累积、上下文膨胀和成本失控。因此需要：
过程约束；
奖励函数；
路由器；
证据充分性检查；
检索轨迹记录；
失败时回退到文本证据。

7. 第五个技术焦点是鲁棒性：GraphRAG 的图并不天然可靠。

多篇论文反对一个隐含假设：只要把文本建成图，检索就会更可靠。

Toward Robust GraphRAG 指出，LLM 构建的 KG 常有两类问题：
虚假噪声边导致检索漂移；
信息不完整导致系统沿着支撑不足的图继续推理，产生检索幻觉。

Beyond Explicit Edges 认为真实图常常稀疏、缺边、有噪声，因此需要“超越显式边”，结合 LLM 导航和 embedding 相似扩展。

GraphRAG under Fire 和 Query-Efficient Agentic Graph Extraction Attacks 则从安全角度指出，GraphRAG 的结构化索引会带来新攻击面：
投毒攻击可能污染社区、摘要和路径；
黑盒查询可能泄露隐藏实体-关系图。

所以该方向的鲁棒性观点是：GraphRAG 不只是检索增强问题，也是图质量、安全、隐私和证据验证问题。

8. 第六个技术焦点是引用、出处和可审计性：最终答案引用的几个来源不足以解释 GraphRAG 的答案。

Why Neighborhoods Matter 提出一个重要观点：代理式 GraphRAG 的答案可能受到“访问过但未引用”的节点和邻居结构影响。因此只看最终引用是否支持答案是不够的，还要看完整遍历轨迹、被访问节点、未引用证据和图结构上下文。

法律类论文进一步强调，法律 RAG 不应只是相似文本检索，而应保留：
条款层级；
时间版本；
修订行为；
法律效力范围；
可复现查询过程。

这类论文的共同观点是：GraphRAG 的可解释性不能停留在“给出引用”，而要走向“检索过程可审计”。

9. 第七个技术焦点是效率和成本：GraphRAG 最大现实障碍之一是建图和检索太贵。

很多论文都在解决成本问题：
HELP 用 HyperNode 扩展和路径定位提升检索速度；
Core-based Hierarchies 用线性 k-core 替代不稳定社区发现；
GraphRAG-Router 学习在不同 GraphRAG 和 LLM 之间做成本感知路由；
ReMindRAG 强调用低成本 LLM 指导 KG 遍历；
GraphRAG on Consumer Hardware 测试本地小模型能否完成医疗 EHR schema GraphRAG；
Towards Practical GraphRAG 关注企业规模建图与混合检索；
Do We Still Need GraphRAG 和 RAG vs GraphRAG 讨论何时应避免过度使用图。

因此，一个务实共识是：GraphRAG 系统需要路由机制。简单问题走普通 RAG，复杂结构问题才走图检索。

10. 第八个技术焦点是评估：现有评测不足以判断 GraphRAG 是否真的有效。

GraphRAG-Bench、When to use Graphs in RAG、How Significant Are the Real Performance Gains、RAG vs GraphRAG 等论文都指出，传统 QA benchmark 主要测文本检索，不足以评估图的价值。

更合理的评估应覆盖：
图构建质量；
证据召回；
上下文相关性；
路径或子图是否覆盖推理链；
答案事实性；
幻觉率；
延迟与成本；
检索轨迹可解释性；
不同问题复杂度下的收益。

这个方向正在从“最终答案分数”转向“建图-检索-组织-生成-引用”的全链路评估。

11. 第九个技术焦点是领域化：GraphRAG 的最强落点不是开放闲聊，而是高约束领域知识系统。

这些论文覆盖了法律、医疗、EHR schema、生物医学、等离子体模拟、自动驾驶安全、ORAN 通信标准、客服、教育评分、数学教材、监管合规等领域。

这些场景有共同特征：
术语密集；
实体关系明确；
证据分散；
错误代价高；
需要可追溯解释；
普通语义相似度检索容易漏掉关键结构。

因此，GraphRAG 在这些场景中更像“领域知识操作系统”的检索层，而不是普通聊天机器人的补丁。

12. Task 1 方向的总体判断

Task 1 的 69 篇论文构成了一个较清楚的学术谱系：

从“把文档建成图”发展到“如何建对图”；
从“检索文本块”发展到“检索证据子图”；
从“一次性检索”发展到“代理式多轮图探索”；
从“答案准确率”发展到“证据充分性、引用忠实性、鲁棒性、安全性和成本”；
从通用 GraphRAG 发展到医疗、法律、教育、工业、自动驾驶等领域系统。

这个方向最核心的研究问题可以概括为：
如何让 LLM 在外部知识图谱上进行可控、可解释、低成本、鲁棒的上下文探索与证据补全。

二、Task 2：多代理 GraphRAG / KG-RAG 方向的主要学术观点

Task 2 中与 LLM GraphRAG 直接相关的 4 篇论文是：
1. Divide by Question, Conquer by Agent: SPLIT-RAG with Question-Driven Graph Partitioning
2. RAGulating Compliance: A Multi-Agent Knowledge Graph for Regulatory QA
3. Advancing Shared and Multi-Agent Autonomy in Underwater Missions: Integrating Knowledge Graphs and Retrieval-Augmented Generation
4. Graph Counselor: Adaptive Graph Exploration via Multi-Agent Synergy to Enhance LLM Reasoning

1. 多代理 GraphRAG 的核心不是“多叫几个 LLM”，而是把图检索任务拆给不同角色或子图。

SPLIT-RAG 的观点很明确：大型知识图谱上的单体检索会导致延迟、无关上下文和多跳推理碎片化。因此它按问题类型和图结构做分区，再把不同子图交给类型专门化代理处理。

这代表一种多代理 GraphRAG 观点：
代理分工应由问题类型、子图结构和领域知识决定，而不是任意设置“专家角色”。

2. 多代理系统中的知识图谱承担共享上下文和可审计记忆的作用。

RAGulating Compliance 面向监管问答，使用多代理构建和维护三元组知识图谱，再结合 RAG 回答法规问题。这里图不仅是检索索引，也是多个代理之间共享、验证和导航的公共知识层。

该方向强调：
监管、法律、合规等高风险场景不能只依赖自然语言上下文；
三元组和证据文本应能回溯；
不同代理的抽取、检索、验证和回答应围绕同一个结构化知识层协同。

3. 在具身/机器人场景中，GraphRAG 不只是问答工具，而是任务规划和自治决策的知识支撑。

水下任务论文把知识图谱、RAG 和 LLM 结合到多 AUV 任务中。它关注的是多个机器人在复杂水下环境中如何理解任务、共享知识、规划行为并做自治决策。

这说明多代理 GraphRAG 可以从“文档问答”扩展到“行动系统”：
图表示环境、对象、任务和约束；
RAG 提供任务相关知识；
LLM 负责解释、规划和生成行为；
多代理系统负责分布式执行。

4. 多代理 GraphRAG 需要解决“图结构与语义意图不一致”的问题。

Graph Counselor 指出 GraphRAG 的一个关键问题：图上的连接不一定等于问题所需的语义推理路径。它用多个代理协同，包括规划、执行、自反思等模块，来做自适应图信息抽取和多角度校正。

这个观点与 Task 1 的鲁棒性研究一致：图结构不是天然正确答案，代理需要根据问题意图动态选择、校正和反思检索路径。

5. Task 2 的总体判断

Task 2 的 4 篇论文共同说明，多代理 GraphRAG 的研究重点在于：
如何把图切分给不同代理；
如何让代理围绕共享知识图谱同步上下文；
如何让不同代理分别负责抽取、检索、推理、验证和行动；
如何避免多代理带来的上下文膨胀、无关协作和不可审计输出。

与普通多代理 LLM 论文相比，这 4 篇更接近“图结构驱动的多代理上下文同步与任务分解”。

三、两个任务合起来看的学术脉络

1. GraphRAG 正在从“检索方法”变成“上下文工程架构”。

早期 RAG 主要关心怎么找文本块；GraphRAG 论文则把问题扩大到：
知识如何建模；
上下文如何组织；
证据如何连接；
检索如何迭代；
生成如何被约束；
引用如何审计；
系统如何扩展和防攻击。

2. 图的价值来自“结构约束”，不是来自“图”这个形式本身。

如果图只是 LLM 随便抽出来的一堆实体边，它可能比普通 RAG 更差。
图真正有价值，是因为它能表达：
层级；
因果；
时间；
组成；
法规修订；
医学概念；
表结构；
任务依赖；
跨文档证据关系。

3. 未来主线很可能是“动态、任务驱动、可验证”的 GraphRAG。

静态预构建图会继续存在，但前沿论文已经明显转向：
查询驱动即时建图；
检索过程强化学习；
图检索路由；
证据充分性检查；
语义-结构混合扩展；
多代理分工；
领域本体约束；
全链路评估。

4. 对实际研究选题的启发

如果要在这个方向继续做研究，比较有价值的问题包括：
如何判断一个问题是否值得走 GraphRAG；
如何低成本构建可用图，而不是昂贵完整图；
如何在不完美图上避免检索漂移；
如何设计子图级、路径级、轨迹级证据评价；
如何把图检索与多代理上下文同步结合；
如何让 GraphRAG 在法律、医疗、工业等高风险领域可审计部署。

简短结论：
Task 1 主要回答“LLM 如何利用图结构完成上下文探索和证据补全”；
Task 2 主要回答“多个 LLM 代理如何围绕图结构分工、同步和协作”。
两者合起来看，LLM GraphRAG 的核心学术方向不是简单的知识图谱问答，而是结构化上下文工程：用图组织知识，用检索选择证据，用 LLM 推理生成，用审计机制控制风险。
